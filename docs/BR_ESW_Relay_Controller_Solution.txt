BR ESW Relay Controller Solution
================================

---------------------------------

                                       docs/architecture.md — state diagram

1. Overview
-----------

Relay controller for up to 10 instances. Each relay uses one DPO output and one
DI feedback input.

Layers:
  - main.c / scheduler — config, requests, periodic task
  - relay_control — controller FSM + relay_instance per relay
  - relay_io — DPO/DI access (sim on host, GPIO on target)

Fault detection runs every task cycle. Mismatch for more than 30 ms latches
WELDED (command OPEN, feedback closed) or CONSTANTLY OPEN (command CLOSE,
feedback open). Any fault puts the controller in ERROR: all relays open, CLOSE
requests blocked until re-init.


2. Assumptions
--------------

  - DI HIGH = contact closed, DI LOW = open
  - DPO HIGH energizes coil, DPO LOW de-energizes
  - NO: de-energized open, energized closed
  - NC: de-energized closed, energized open
  - ERROR cleared only by RelayController_Init()
  - Startup drives all relays OPEN
  - RelayIo_Init() once before controller init


3. System context
-----------------

Up to 10 relays. Demo uses two: relay 0 NO, relay 1 NC (see kRelayConfig in
main.c).

5. NO / NC DPO mapping
----------------------

  Type    CLOSE       OPEN
  NO      DPO HIGH    DPO LOW
  NC      DPO LOW     DPO HIGH

  Command   Expected feedback
  CLOSE     DI HIGH
  OPEN      DI LOW


6. Faults
---------

  Welded           OPEN command, feedback stays closed > 30 ms
  Constantly open  CLOSE command, feedback stays open > 30 ms

Faults are latched until re-init.


7. Error reaction
-----------------

  1. Controller -> ERROR
  2. All relays forced OPEN
  3. CLOSE requests ignored until re-init


8. Timing and scheduling
------------------------

  Task period       5 ms (kRelayTaskPeriodMs)
  DPO settle        1 ms (simulation)
  Fault window      30 ms (6 consecutive mismatch cycles at 5 ms period)

Each call to Scheduler_RunTask() is one 5 ms task period: it runs
RelayController_Process() once. On the host demo, main() calls it in a loop as
fast as the CPU allows — each loop iteration is a logical tick (tick 0 = 0 ms,
tick 1 = 5 ms, and so on), not real wall-clock delay.

On the target MCU, the same Scheduler_RunTask() would be triggered by a hardware
timer or a periodic RTOS callback every 5 ms.

Fault timing inside relay_instance uses cycle counters from compile-time constants
in common.h (no runtime ms timers in the controller).


9. Build and demo
-----------------

  make
  ./build/relay_controller_demo

Demo runs two scenarios (controller re-init between them):

  Scenario 1 — WELDED (relay 0, NO):
    - tick 5: close both relays
    - tick 25: weld relay 0, command OPEN -> WELDED -> ERROR (~tick 31)

  Scenario 2 — CONSTANTLY_OPEN (relay 1, NC):
    - tick 5: close both relays
    - tick 25: stuck-open relay 1, command CLOSE -> CONSTANTLY_OPEN -> ERROR

  ./build/relay_controller_demo > simulation_log.txt
